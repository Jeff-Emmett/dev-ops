# Runbook — rotate `gemini-api-key`

Shared Google Generative Language API key (`AIza…`). Consumed by the gemini
MCP server, `gemini-proxy`, and the rMail `gemini-flash` agent.

**Status:** EXPOSED in a Claude session on **2026-06-13** (a debug echo printed
the live key into chat → it is in Anthropic's logs). Treat as compromised.
Rotate. (Memory: `rotate-gemini-api-key`.)

## Why this is semi-manual

Google API keys can only be regenerated by a human in the Cloud Console — there
is no programmatic mint API. So exactly one step is manual; everything after it
is automated by `rotate-gemini-api-key.sh`.

## Step 1 — regenerate (manual, only you can do this)

1. Open https://console.cloud.google.com/apis/credentials
2. Find the Gemini / Generative Language API key.
3. Click **Regenerate key** (preferred — it invalidates the old value
   immediately, so no separate revoke is needed). If you instead **create a
   new key**, you must delete the old one after Step 2 verifies.
4. Keep the key **restricted to the Generative Language API**.
5. Save the new value into a temporary mode-600 file, e.g.:
   ```bash
   umask 077; printf '%s' 'AIza...NEWKEY...' > /tmp/gemini.new
   ```
   Do **not** paste the key into the Claude chat — that re-leaks it.

## Step 2 — propagate (automated)

```bash
cd ~/Github/dev-ops
./security/rotate-gemini-api-key.sh --dry-run --key-file /tmp/gemini.new   # preview
./security/rotate-gemini-api-key.sh --key-file /tmp/gemini.new             # apply
shred -u /tmp/gemini.new                                                   # destroy the temp copy
```

The script never accepts the key as a positional argument (that would land in
shell history / `ps` / logs). It:

- updates the local MCP key file `/home/jeffe/.private/claude-mcp/gemini-key`
  (backs up the old one),
- upserts Infisical `rspace:/prod/GEMINI_API_KEY` (write-capable identity in
  Netcup `/opt/rspace-online/.env`),
- upserts Infisical `claude-ops:/ai/GEMINI_API_KEY` **only if** you point
  `CLAUDE_OPS_ENV` at a write-capable identity env file — otherwise it warns
  and you update that one canonical copy in the Infisical UI,
- restarts server-side consumers (`gemini-proxy` if present),
- marks `last_rotated` in `secrets-inventory.yaml`.

### claude-ops canonical copy

Per the rotation memory, the `rspace` identity can write `rspace` but the
ops/claude-ops read identity **cannot write** `claude-ops`. So unless you supply
a write-capable identity via `CLAUDE_OPS_ENV=...`, update
`claude-ops:/ai/GEMINI_API_KEY` by hand at https://secrets.jeffemmett.com after
running the script.

## Step 3 — reload + verify

1. **Restart Claude Code** (or `/mcp` reconnect) so the gemini MCP re-reads the
   key file. Smoke test: `mcp__gemini__gemini_chat` with a trivial prompt.
2. Confirm `gemini-proxy` came back healthy on Netcup (`docker logs gemini-proxy`).
3. If you used "create new" rather than "regenerate", **delete the old key** in
   the Cloud Console now.
4. Confirm both Infisical paths hold the new value.

## Notes

- Other places that might read `GEMINI_API_KEY`: grep server `.env` files and
  `~/.claude.json`. The local box reads it from the key file, not inline.
- This runbook pairs with the `propagation_script` field in the
  `gemini-api-key` inventory entry.
